## ⚠️ 缺點 / 風險（嚴格版，依嚴重度）

### 🔴 P0 — 資料安全根因（同步競態）
- **問題**：同步是「逐 key 比時間戳、新的贏」。若某裝置在雲端資料**還沒拉下來前**先用空白/預設狀態存了一次某個 key（拿到很新的時間戳），下次同步這個「空白但較新」的版本會**反向蓋掉雲端正確資料**，再擴散到所有裝置。
- **證據**：習慣清單消失事件就是這個機制中獎（已上自癒補丁，但那是症狀解）。
- **威脅**：理論上能傷到任何資料，不只習慣。對「要賣」是致命傷——掉資料 = 信任歸零。
- **為何今晚沒動**：修法（拉取優先閘門）會改動登入/同步啟動流程，**需要你醒著用真帳號測登入**，否則可能擋掉朋友現在的存取。這是唯一一個「太大、不該盲修」的。

### 🟠 P1 — Supabase 多實例競態（今晚已修 ✅）
- `supa()` 是 async，在 null 檢查與 createClient 之間有 await，並發首次呼叫會各自建 client → "Multiple GoTrueClient instances" 警告 + auth/同步未定義行為。
- **已改**：記住 Promise 而非 client，保證全程只建一個。clean load 後警告歸零。

### 🟡 P2 — 時程（Schedule）桌機頁偏空
- 桌機版時程頁只有頂部一條週甘特，下方大片留白，資訊量低、看起來「沒做完」。
- 建議：要嘛把當日 timeline 也搬進來做成「週+日」雙欄，要嘛跟週分頁合併。**屬於重設計，留你決定。**

### 🟡 P2 — 無障礙（a11y）
- 大量純 icon 按鈕（▶ ✓ − ＋ ⧉ ✍️ 🔔…）沒有 aria-label，螢幕報讀器使用者讀不到。
- **今晚已修頂部導覽**（🔍 搜尋 / ⚙ 設定）；其餘是廣面改動，列為 backlog。

### 🟢 P3 — 雜項
- 桌機資訊密度對新手可能偏高（但這是「給認真的人用」的取捨，不一定要改）。
- `<title>` 靜態（SEO/分享 OK，不急）。

## 🎯 給「要賣」的策略判斷（重申）

產品完成度已經不是問題，**功能再加也不會讓它更好賣**。yike 的價值不在「防守一個 US$12 planner」，在於：
1. **build-in-public 素材**：「我用 AI 一個週末做出有質感的第二大腦工具」本身就是品牌最強 demo。
2. **電子報名單磁鐵**：影片 CTA 收訂閱（M2 主事 ≥100），不是賣 app。

能被快速複製**無所謂**——複製者拿走 app，拿不走你的頻道、方法論、觀眾。
真正的風險是：**把 yike 當成要防守的產品無限打磨，吃掉 M2 的時間。** yike 該服務品牌，別變成品牌。
